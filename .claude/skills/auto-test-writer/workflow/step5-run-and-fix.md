# Step 5 - 运行并修复

## 运行

使用 step2 记录的运行命令执行**本次新增/修改的测试文件**（优先只跑增量，加快反馈）：

- pytest: `pytest tests/foo/test_bar.py -x -q`
- vitest: `npx vitest run src/foo/bar.test.ts`
- jest:   `npx jest src/foo/bar.test.ts`
- go:     `go test ./foo/...`

增量通过后，再跑一次**全量**测试，确认没有回归：
`pytest -q` / `npx vitest run` / `go test ./...`

## 失败处理循环

若有失败，按如下优先级诊断：

1. **测试本身写错了**（断言期望值算错、mock 设置不当）→ 修测试
2. **源码 bug 暴露**（测试正确揭示了代码缺陷）→ 修源码，但改动范围保持最小
3. **环境/依赖缺失**（import error、fixture 未注册）→ 补齐
4. **无法确定** → 先 Read 失败的测试和被测源码，再做判断；仍不确定则停下来告知用户，不要瞎猜

修改后**重跑**。重复直到全部绿灯。

## 硬约束

- **禁止**通过删除测试、放宽断言、`skip` / `xfail` 来"让测试通过"
- **禁止**捕获并吞掉异常以绕过失败
- **禁止**在未搞清失败原因的情况下连续盲目修改超过 2 次——此时应停下来向用户求助
- 测试全部通过前，**不得**进入 Step 6，**不得**执行 `git commit`
