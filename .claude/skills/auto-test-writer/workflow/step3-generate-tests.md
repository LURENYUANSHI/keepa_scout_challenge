# Step 3 - 生成测试用例

## 目标

为 step1 变更清单中的**每一个**新增/修改函数，生成至少 3 个 case。

## 每个函数的强制覆盖

| 类别 | 含义 | 示例 |
| ---- | ---- | ---- |
| 正常路径 | 典型合法输入，断言返回值/副作用符合预期 | `add(2, 3) == 5` |
| 边界值 | 临界输入 | 空列表、0、最大长度、None、边界日期 |
| 异常输入 | 非法输入应触发错误或返回错误标识 | 类型错误、越界、网络失败 |

**缺一不可**。对纯 UI 组件等无明确"异常输入"的情况，用"快照 / 属性缺失 / props 类型错误"代替。

## 生成原则

1. **先 Read 源码**：生成前必须 Read 被测函数所在文件完整内容，理解依赖和副作用
2. **真实断言**：断言应基于函数的实际逻辑，不要写 `assert result is not None` 这种无意义断言
3. **隔离外部依赖**：对数据库/网络/文件系统调用，使用框架对应的 mock（pytest `monkeypatch`、vitest `vi.mock`）；但**不要过度 mock**——能用真实简单数据就用真实数据
4. **一个 case 一个 test 函数**：命名清晰，例如 `test_parse_config_happy_path`、`test_parse_config_empty_file`、`test_parse_config_invalid_yaml`
5. **自包含**：每个测试函数独立可运行，不依赖其他测试的执行顺序
6. **遵循项目已有风格**：先 Read 1-2 个现有测试文件，模仿其 fixture、import、断言风格
