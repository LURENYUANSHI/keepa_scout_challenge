# 风险等级定义

| 等级 | 含义 | 典型例子 |
| ---- | ---- | -------- |
| **Low** | 无或仅风格微瑕，不影响正确性、可维护性、兼容性 | 1-2 处可优化的命名；全部检查通过 |
| **Medium** | 存在真实缺陷或可维护性问题，不修会在近期制造麻烦 | 多余的无关改动；少量 magic number；2 处重复代码；内部调用者需要同步修改的签名变化；修改旧分支未覆盖 |
| **High** | 缺陷会很快导致 bug 或下游破坏 | 吞异常；外部调用方会失败的签名变化；硬编码 URL/路径；新增代码未覆盖；静态分析在依赖模块报新错 |
| **Critical** | 直接危险，绝对不能进 commit | 硬编码密钥 / token / 密码；删除数据库字段但未迁移；schema 不兼容变更且无兼容层；删除被外部消费的公共 API；业务逻辑错误会导致数据损坏 |

## 升级规则

- 单项出现多个 Medium → 该子项升级为 High
- Critical 子项单独存在就让总风险直接 Critical
- 破坏性变更 + 无调用方全量更新 → 至少 High

## 降级规则（严格限制）

- 不得因为"时间紧"降级
- 不得因为"下次再修"降级
- 只有当 finding 在重新分析后被证明是**误判**时才能撤销该条 finding，而不是降级
